# プロポーザル: Rustコードベース全体リファクタリング（レイヤードアーキテクチャ × SOLID原則）

## 概要

KatanAの4レイヤー（core, linter, platform, ui）に対し、保守性と拡張性を根本的に高めるためのリファクタリングを実施する。肥大化したファイルの分離、レイヤードアーキテクチャの再設計、SOLID原則違反の解消を通じて、今後の機能追加・バグ修正を高速かつ安全に行える基盤を構築する。

> ※ 旧 `v0-8-4-refactoring`（Settings分離設計）は本プロポーザルに統合済み。

## 背景・課題

### 1. ファイルの肥大化

テスト除外で200行以上のファイルが**43ファイル**存在し、特にUIレイヤーは深刻である：

| ファイル | 行数 | レイヤー | 主な問題 |
|---|---|---|---|
| `shell_ui.rs` | 5,118 | ui | 46個のfree関数が1ファイルに集中。メニュー・ステータスバー・ワークスペース・タブ・検索・ToC等の全UIが同居 |
| `shell.rs` | 3,144 | ui | KatanaApp構造体に40以上のメソッド。ファイル操作・ダウンロード・更新・エクスポート・プレビュー管理が混在 |
| `preview_pane.rs` | 1,816 | ui | ダイアグラムレンダリング・画像キャッシュ・フルスクリーン・スクロール同期が同居 |
| `settings_window.rs` | 1,666 | ui | 全設定タブのUI描画が1ファイル |
| `preview_pane_ui.rs` | 1,270 | ui | セクション描画・フルスクリーンモーダル・SVG表示が混在 |
| `i18n.rs` | 1,092 | ui | 翻訳文字列定義とロジックが混在 |
| `rust.rs` (rules) | 969 | linter | 6個のVisitorが1ファイルに全展開 |
| `widgets.rs` | 948 | ui | Modal, ComboBox, Toggle, ColorPicker が1ファイル |
| `settings.rs` | 653 | platform | 分離開始済みだが、旧ファイルが残存 |

### 2. SOLID原則の違反

- **SRP（単一責任）違反**: `KatanaApp` が「ファイル操作」「ダウンロード」「更新チェック」「エクスポート」「プレビュー管理」「ワークスペース管理」を一手に担う God Object
- **SRP違反**: `AppState` に57個のフィールドが集中（UI状態、ワークスペース状態、検索状態、設定状態、更新状態等が混在）
- **OCP（開放閉鎖）違反**: `shell_ui.rs` のメニュー構成やタブ追加が既存コードの直接修正を必要とする
- **ISP（インターフェース分離）違反**: `PreviewPane` がMarkdownレンダリング・ダイアグラム管理・画像キャッシュ・フルスクリーン表示の全責務を持つ
- **DIP（依存性逆転）の未整備**: レイヤー間の依存が具象型への直接参照のみ（trait抽象なし）

### 3. レイヤー間の現在の依存関係

```
katana-core    → (外部クレートのみ)
katana-linter  → (外部クレートのみ、katana-coreに依存しない独立レイヤー)
katana-platform → katana-core
katana-ui       → katana-core, katana-platform
```

## 本提案による解決策

1. **4レイヤーごとの階層設計**: 各レイヤー内をサブモジュールに分割し、責務を明確化
2. **肥大化ファイルの分離**: 200行超のファイルを責務単位で分離（ast_linterで200行制限を強制）
3. **SOLID原則の適用**: God Object の解体、責務の明確な分離
4. **共通処理の検討**: 4レイヤーに共通する処理パターンの切り出し可否を分析

## 目標

- すべてのソースファイル（テスト除外）が200行以下
- 各レイヤーの内部構造がレイヤードアーキテクチャに準拠
- SOLID原則に基づいた責務分離が完了
- `make check` が通る状態を維持
- ast_linterによる200行制限の機械的な強制

## 将来的な検討事項

- ast_linterでレイヤードアーキテクチャの階層制約を機械的にチェックする仕組み（階層設計完了後に検討）
