## Why

初回インストール後、ユーザーが新バージョンをダウンロードするために毎回ブラウザでReleasesページを訪問し、DMGを再ダウンロード・マウント・ドラッグする手順は煩雑であり、ユーザー離脱の原因になる。

アプリ内で自動更新を完結させることで、DMGは初回インストール時のみ必要となり、以降はアプリ単体で最新版を維持できる。

## What Changes

### アプリ内自動更新機能

1. **起動時バージョンチェック**: アプリ起動時にバックグラウンドで GitHub Releases API (`/repos/{owner}/{repo}/releases/latest`) を確認し、新バージョンがあればダイアログで更新を促す（CHANGELOG の差分サマリを表示）
2. **更新通知UI**: 新バージョンが利用可能な場合、更新内容の概要とともにダイアログで通知。「今すぐ更新」「後で」「このバージョンをスキップ」の選択肢を提供
3. **ダウンロード＆置換**: 自動更新用に配布される `.zip` （`.app` バンドルを圧縮したもの）をダウンロードして一時ディレクトリに展開。その後、自身を終了して一時的なバックグラウンドスクリプト（Relauncher）に処理を引き継ぎ、`.app` のアトミック置換と隔離解除（`xattr -cr`）を実行させる
4. **再起動プロンプト**: 更新完了後、ユーザーに再起動を促す（自動再起動オプション付き）
5. **チェック頻度設定**: 起動時チェックの有効/無効切り替え、チェック頻度の設定（デフォルト: 毎回起動時）

### CI/CD変更

1. **Releaseワークフロー拡張**: 既存の DMG に加え、ビルドされた `.app` バンドルを圧縮した `.zip` アーティファクト（自動生成されるソースコードのzipとは異なる）を生成・アップロードする
2. **バージョン情報エンドポイント**: GitHub Releases APIを利用（追加インフラ不要）

## Technical Considerations

### 置換の原子性

- 新しい `.app` を一時ディレクトリにダウンロード＆展開
- 魔法のようなインプレース置換を避け、1行の bash スクリプト（Relauncher）をバックグラウンド実行してKatanA自身は即座に終了する
- スクリプト側で「旧アプリの待機・削除 -> 新アプリの移動 -> `xattr -cr` -> テンポラリファイル（ダウンロードしたZIP等）のクリーンアップ -> `open` 起動」をアトミックに行う

### 権限
- ユーザーが自分でインストールした `/Applications/` 配下は書き込み権限あり
- システム管理者がインストールした場合は権限昇格が必要（将来対応）

### ローカル検証可能性 (Local Testability)
- いきなり本番リリース（GitHub Releases）でテストするリスクを避けるため、APIエンドポイントのURLを環境変数等（例: `KATANA_UPDATE_API_URL`）で上書き可能にする
- ローカルに立てたダミーHTTPサーバー（Pythonの `http.server` 等）から偽の更新JSONとダミーの `.zip` を配信し、本番リリース前に手元で完全に自動更新の挙動テスト（E2E検証）が完結する設計とする

### セキュリティ

- ダウンロード元はGitHub Releases（HTTPS）のみ
- チェックサム検証やサイズ検証を実装
- ad-hoc署名を維持（`xattr -cr` により隔離属性を剥奪するため、無課金でもアプリ実行が可能）

## Capabilities

### New Capabilities

- アプリ内バージョンチェック
- アプリ内ダウンロード＆自動更新
- 更新通知UI

### Modified Capabilities

- Releaseワークフロー（自動更新用 `.zip` 配布の追加）

## Impact

- `katana-core`: 更新チェッカーモジュール追加
- `katana-ui`: 更新通知UI追加
- `.github/workflows/release.yml`: .zip アーティファクト作成とアップロード追加
- 新規依存: ZIP解凍用に `zip` (または `zip-extract`) クレートの追加が必要（またはOSコマンド `unzip` の利用）
