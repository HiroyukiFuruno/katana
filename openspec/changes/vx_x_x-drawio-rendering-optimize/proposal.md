# Proposal: Draw.io レンダリングの最適化と完全対応

## 背景と課題
現在、KatanaのDraw.ioレンダリング実装 (`drawio_renderer.rs`) は、XMLをパースして独自にSVGを出力する簡易的な実装 (MVP) となっています。
この実装は基本的な矩形や直線には対応していますが、今回サンプルとして提供された `aws.drawio` に含まれるような、高度なステンシル (`mxgraph.aws4`) や複雑なグループ化・ルーティングには対応しきれず、表示が崩れたり描画されなかったりする「実装の不備」が生じています。
Draw.ioのすべての仕様をRustのSVGジェネレータで再実装するのは非現実的であるため、正確にレンダリングするための抜本的なアプローチの変更が必要です。

## 目的
- 複雑な `drawio` 形式（AWS構成図など）のファイルを、公式ツールと同等の完全な精度でプレビューペインにレンダリングする。
- ユーザー体験を向上させるため、Markdownの画像埋め込み構文 (`![alt](file.drawio)`) による直接レンダリングをサポートする。

## 解決アプローチ (計画)

### 1. `drawio` ファイルの完全レンダリング対応
Mermaidのレンダリング (`mermaid_renderer.rs` が `mmdc` を呼び出す形式) と同様に、Draw.io公式のCLI機能を利用してバックグラウンドで画像（PNGまたはSVG）へエクスポートするアーキテクチャに変更します。
- **macOS環境でのCLI利用**: Draw.ioのデスクトップアプリがインストールされている場合、標準的なアプリアプリケーションパス (`/Applications/draw.io.app/Contents/MacOS/draw.io`) にCLIが存在します。これをサブプロセスとして呼び出し、ヘッドレスで図をエクスポートします。
- **依存関係なし（Docker 不要）**: Draw.io Desktop は Electron (Chromium バンドル) アプリであり、Docker・Node.js・Java などの外部ランタイムは一切不要です。Mermaid CLI（Node.js が必要）と比較してセットアップが非常にシンプルで、「Draw.io Desktop がインストール済みか否か」の1点のみを確認すればよいです。
- **処理フロー**: `![alt](file.drawio)` または ````drawio` ブロックの入力を受け取り → 一時ファイルとして保存 → `draw.io` CLI を呼び出しPNG化 → プレビューのeguiテクスチャとして画面に描画。
- **非同期・キャッシュ考慮**: CLI呼び出しはコストがかかるため、入力ファイルやテキストのハッシュを利用して生成結果をキャッシュする仕組みを設けます。

### 2. Markdownでの `.drawio` ファイル直接マウントへの対応
ユーザーからの質問：
> 「imageのようにdrawio拡張子のファイルを添付したら、そのまま表示するようなことも考えているんですが、markdownの標準仕様から外れますかね？」

**回答**:
Markdownの標準仕様 (CommonMark) では、画像構文 `![alt](path)` でサポートされるフォーマットについて厳密な制約は設けておらず、レンダラー（ブラウザなど）に依存するとされています。
この構文を拡張して、`.drawio` や `.puml` などを直接画像として解決しレンダリングするアプローチは、**最近の高機能Markdownエディタ（Obsidian, VS Codeなど）では標準的で広く受け入れられているベストプラクティス**です。標準仕様を明確に破壊するものではありません。
むしろ、コードブロック（````drawio`）に巨大なXMLを直書きするよりも、外部ファイルを画像としてマウントする方がMarkdownとしての可読性が圧倒的に高まるため、**強く推奨される拡張**です。

## 範囲外 (Out of Scope)
- 組み込みのエディタ上でのDraw.io図形の直接編集機能（あくまでプレビューレンダリングに留める）。

## リスクと制限事項
- ローカル環境に Draw.io Desktop アプリが存在する必要があるため、未インストールの場合は Mermaid と同様に「未インストール」のガイダンス表示やダウンロードリンクへ誘導するUI実装 (`RenderedSection::NotInstalled`) が必要です。
