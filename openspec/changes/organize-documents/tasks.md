## 1. Document Cleanup inside `docs/`

- [ ] 1.1 `docs/` ディレクトリ内の既存のドキュメントをレビューし、リストアップする。
- [ ] 1.2 今後の計画、ロードマップ、仕様書にあたる企画的なドキュメントを特定して削除（または移動）する。
- [ ] 1.3 必要な撤去済みドキュメントは、新たな関連仕様・プロポーザルとして `openspec` 側へ再作成する。

## 2. Setting Up Parallel Documentation（並行ドキュメント）

- [ ] 2.1 `docs/` に残る説明的なドキュメントについて、元のファイルと日本語ファイル `.ja.md` （例： `README.md` と `README.ja.md`）の形式による命名規則を標準化する。
- [ ] 2.2 `docs/` に残響するすべての日本語ドキュメント要素を翻訳し、英語の標準版として作成する。
- [ ] 2.3 元の日本語ドキュメントを正確な `.ja.md` 拡張子へと明示的にリネームする。
- [ ] 2.4 プロジェクトルートに、`README.md` の日本語並列版となる、`README.ja.md` を作成する。

## 3. General Repository Translation

- [ ] 3.1 `docs/` ディレクトリの外側にある一般のリポジトリドキュメント群（ `/README.md`, `CONTRIBUTING.md` など、ただし `*.ja.md` の日本語版を除く）をリストアップする。
- [ ] 3.2 抽出されたマークダウンドキュメントを、プライマリ言語として英語へ書き換える・翻訳する。
- [ ] 3.3 翻訳のプロセスから、`i18n` の各種設定、特例である `openspec/` 以下のすべてのディレクトリ設定、および `.gitignore` パターン化された除外ファイルが適切にブロック・反映されていることを保証する。

## 4. Codebase Comments Translation (Phased Approach)

- [ ] 4.1 プロジェクト内の日本語コメント部分を洗い出すための抽出や検索の手順を確立する（ただし `.gitignore` への合致、`openspec/`、`i18n` ディレクトリなどを完全に検索対象外とする設定を含む）。
- [ ] 4.2 第1フェーズ: 共有コアモジュール（例: 共通ユーティリティや基盤など）に存在する特定のコメントを翻訳する。
- [ ] 4.3 第2フェーズ: 主力機能クレート（例: `crates/katana-ui` など）内の特定のコメントを英語に翻訳する。
- [ ] 4.4 第3フェーズ: 残存するその他のソースファイルを英語翻訳するとともに、技術用語の脈絡やニュアンスに破綻がないかを確認・レビューする。
