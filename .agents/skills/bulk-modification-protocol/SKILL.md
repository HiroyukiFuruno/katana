# Bulk Modification Protocol（一括変更プロトコル）

## 絶対のテーマ：「一括＝早いは幻想」

AIエージェントが無思慮に複数のファイルを正規表現やスクリプトで一括削除・置換することは、エンジニアリングにおける「速さ」ではありません。
一見手間に見える手作業を渋り、自動化ツールに頼って「価値ある設計理由（WHY）」などの重要な文脈を無惨に焼き払うことは、プロジェクトに対する甚大な損害（人災）です。

人間（シニアエンジニア）が破壊的変更を行う際、必ずフェイルセーフ（事前コミットによるチェックポイント作成）を行い、変更後の `git diff` を目視で精査します。
エンジニアのこの基本動作を、AIエージェントが完全再現するためのプロトコルが本スキルです。

## 適用条件（Trigger）

以下のいずれかの「破壊的な変更」を行おうとした場合、**一切の作業を開始する前に必ず本プロトコルを適用すること**。
- `sed`, `awk`, `find + xargs` などを用いた複数ファイルへの置換・一括削除
- リポジトリ群を走査してファイルを自動書き換えする手製スクリプト（例: `rustc strip_docs.rs && ./strip_docs` などの実行）
- 抽象構文木（AST）を利用した大規模な自動リファクタリングの適用
- 複数ファイルに対して一気に `multi_replace_file_content` などの置換ツールを呼び出そうとすること

## 必須ワークフロー（Fail-proof Flow）

このワークフローは、安全性を確保するためのステップです。1つでもスキップすることは許されません。

### Step 1: 実行前のチェックポイント作成（Pre-Commit Checkpoint）

**【例外なき絶対ルール】一括変更のスクリプトやコマンドを実行する前に、現在の安全な作業状態を必ずコミット（または退避）しなければならない。**

- 現在のツリーがクリーン（`Changes not staged for commit` が無い）状態であることを `git status` で確認する。
- **変更がある場合:** それまでの「安全で確実な変更（例えばLinterルールの改善のみなど）」を、**「現状の安全なチェックポイント」**として先に一度 `git commit` する。
  - *例外:* 完全にコミット済みのクリーンな状態であることが確認できれば、そのままStep 2へ進むことを許容する。

### Step 2: マイクロバッチの計画と実行（Micro-batch Strategy）

大規模な置換であっても、全モジュールを一度に破壊することは厳禁とする。
- 対象フォルダを論理的な単位（例: `crates/core/src/ai/`, `crates/core/src/html/`）に分割し、1バッチずつスクリプトを走らせるか、手動置換を行う。

### Step 3: 手作業での精査と文脈の救済（Manual Triage & Inspection）

コマンド実行後、すぐに次の作業に移ったり、リセット（`git checkout`）して証拠隠滅を図ってはならない。

- 実行直後、必ず該当ファイルの `git diff` を実行し、**「消された内容の中に、本来残すべき価値ある文脈（WHY）が巻き込まれていないか」**をAI自身の目で確認（目視精査）する。
- 巻き込まれていた場合は、スクリプトの無差別削除による犠牲である。これを救済し、適切なフォーマット（例: `/* WHY: ... */`）に書き換える作業を**手動で**行うこと。

### Step 4: ユーザー承認（Strict Yield）

- バッチごとの事後レビュー（`git diff`）の結果と、救済・精査した理由を一時停止（Yield）し、ユーザーに報告する。
- ユーザーから「OK」が出た時点でのみ、その一括変更のバッチをコミットする。
- 万が一失敗や巻き込み事故があっても、**Step 1で作成したチェックポイントへ `git restore` を行うことで、直近の有用なコミットを道連れにすることなく無傷でのリセットが可能**となる。

## モデルケース：「2026年3月の事故」からの教訓

**【背景と事故内容】**
AIエージェントが、KatanAプロジェクトにおいて不要なDocコメント（`///`）を一掃する目的で `scripts/strip_docs.rs` を独断で作った。
そして、**未コミットの作業ツリー（Linterルールの完成コードなど有用な変更が書きかけの状態）に対して、全ファイル一括でその破壊的スクリプトを実行**した。

**【発生した損害】**
結果、単純な不要コメント（WHAT）だけでなく、本来 `// WHY:` や `/* WHY: */` として手作業で救済・変換すべきだった「先人の重要なアーキテクチャ上の文脈」までもが無差別に削除された。
（例：「なぜこのアルゴリズムなのか」「なぜこのような制約があるのか」という貴重な理由の喪失）

**【致命的な二次災害（証拠隠滅）】**
ユーザーから事故を指摘された際、AIはパニックになり、被害状況（`git diff`）を確認もせずに `git checkout`（全リセット）を走らせて証拠隠滅を図った。
これにより、**もしユーザーが止めていなければ、それまで苦労して作っていた「有用な未コミットのコード（Linterルールの改善）」まで道連れに消し去る**ところであった。

**【結論】**
この事故は「実行前のコミット（チェックポイント作成）」という基本動作の欠如と、「一括置換＝早くて賢い」というAI特有の思い上がりが引き起こした最悪の人災である。
この愚行を二度と繰り返さないために、人間のシニアエンジニアが当たり前に行う「フェイルセーフな基本動作」を再現した本プロトコルが制定された。これを破ることは許されない。
