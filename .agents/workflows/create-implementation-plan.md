# Create Implementation Plan Workflow

このワークフローは、機能実装に先立って Antigravity が `implementation_plan.md` アーティファクトを作成する際に必ず順守するプロセスを定義します。

## 1. 言語の自動判定と決定
- まず、対象マシンのロケール（locale）を判定（例: `locale` や `echo $LANG` コマンドを使用）し、計画書は原則としてその言語で作成してください。
- **例外事項 (Exception)**: ユーザーからの依頼文章から言語が明確に特定できる場合（日本語で依頼されている等）は、マシンのロケールに関わらず、**依頼文章の言語を最優先**して計画書を作成してください。

## 2. タスク内容の重複記載の禁止
- 計画のタスク内容やスコープが、すでにプロジェクト管理対象のドキュメント（例: `tasks.md`）や既存の `task.md` アーティファクトに記述されている場合、それらの**具体的内容を implementation_plan.md にコピペ・重複記載してはいけません**。
- 代わりに、「具体的なタスク・スコープについては `tasks.md` を参照すること」などの参照リンク（Ref）を置くにとどめ、ドキュメントの肥大化と二重管理を防いでください。

## 3. TDD (Red -> Green -> Refactor) サイクルの明記
- 実装計画には、KatanAプロジェクトのコーディング規約に基づく基本アプローチとして、必ず以下の内容を開発プロセスとして定義・明記してください：
  - **Red**: 実装を書き始める前に、要件を満たさないことを自動で証明する【失敗するテスト】（Unit Test または UI Integration Test）を作成すること。※UIスナップショット画像比較は使用しない。
  - **Green**: 失敗をアサートしたのち、そのテストを通過させるための【最小限の実装】を追加すること。
  - **Refactor**: グリーン状態を維持しながら、コードのクリーンアップや責務の整理を実施すること。

## 4. UIのフィードバック・ループの明記
- デザインカンプ（Figmaなど）が存在しないプロジェクトの性質上、UIの新規作成・変更を伴う実装計画においては、**「実装したUI（スクリーンショット等）をユーザーに提示し、フィードバックを受けて微調整を行う工程」**が必ず含まれることを計画内で宣言してください。

## 5. ユーザーへの計画提示
- 完成した `implementation_plan.md` は、`notify_user` ツールを使用して適切にユーザーへ提示し、承認（アプルーバル）を得てから実装（EXECUTION）に移行してください。
