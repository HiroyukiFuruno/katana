# 概要 (Task Micro-Cycle)

このワークフローは複数タスクが同時に記述されている環境において、AIエージェントが**「勝手に複数タスクをまとめたり、事前承認なしで一気にコミットまで駆け抜ける暴走行為」を完全に防ぎ、ユーザーの介在・統制のもとで安全にプロジェクトを進めるため**に必須のプロセスを定義します。

# 適用条件
- `tasks.md` に未完了のタスクがあり、実装・修正作業を伴う場合。
- ユーザーから「進めて」などの作業継続指示を受けた場合。

# 厳守すべきマイクロサイクル手順

## 1. 対象タスクの特定とスコープ宣言
- `tasks.md` から今回該当する「単一のタスク単位」（例: 1.5 や 2.2 など）を抽出する。
- 複数のタスクに対してまたがる修正（先回り作業）は絶対に行わない。
- ユーザーに対して「これから行うのは x.x のタスクのみです」と宣言する。

## 2. 実装と自己検証（Self-Review）
- 当該タスクに関する修正（1ファイルの分割、Linterの設定追加など）を行う。
- 実装内容および自己レビューには **必ず `docs/coding-rules.ja.md` を参照・適用** すること。
  - 同時に、このルールファイル内に記述されている制約のうち、AST Linterで静的に弾けるものが存在する場合は、自律的に `katana-linter` 側へのルール実装を行うこと。
- 変更を行ったスコープに対して、Linter (`cargo clippy` 等) やコンパイルテスト (`cargo check` 等) を実行する。
- **他レイヤーへの波及エラーの許容**: Linterを全適用した場合など、他レイヤー/クレートでテストが通らなくなる場合は、本タスクの目的と照らし合わせて「許容範囲（後続フェーズで対応）」と見なすか自己判断し、ユーザーへの報告にその理由を含めること。

## 3. 一時停止とユーザー検証（Strict Yield Control）
- **【重要】** 実装・自己検証が完了した時点で、修正内容（Diffの要旨等）と評価結果を報告し、**ツールチェーンの実行（バッチ処理）を直ちに停止してユーザーの承認を待つこと**。
- ここで決して同一ターン内で `git commit` 等を継続して呼んではならない。

## 4. コミット実行と次のサイクルへ
- ユーザーから「OK」「進めてよし」などの明示的な承認・フィードバックが行われた次のターンで初めて `git commit`（`SafeToAutoRun: false` 若しくは承認済みの確証あり）を行う。
- `tasks.md` を更新し、完了した項目にチェック `[x]` を入れる。
- コミット完了後、次のタスク単位（x.x+1）について再度ステップ1からサイクルを開始する。
